Welcome to HexagonOS Developer Nexus

HexagonOS is a modular, web-based operating system built entirely in HTML, CSS, and JavaScript. It's designed to run in the browser, simulating a desktop environment with windows, taskbar, notifications, widgets, and more. Think of it as a sandboxed universe where apps live in iframes, communicating with the kernel via a secure bridge. No native installs—everything is web-native, lightweight, and extensible.

Why build for HexagonOS? It's a playground for devs to create apps that feel native but are pure web tech. Users get a seamless experience: drag windows, minimize, multitask, and personalize. As a dev, you tap into system features like storage, services, and themes without direct DOM access—everything's mediated for security.

What You'll Learn Here

From zero to hero: Start with basics, build a full app, integrate with system APIs, and submit to the app store. No prior OS dev experience needed—just web skills.

Key Philosophy

Modular, secure, and user-first. Apps are isolated, storage is distributed to avoid limits, and everything persists via localStorage.

Core Rules & Architecture

HexagonOS enforces isolation for stability. Your app runs in an iframe, so you can't touch the host DOM directly. Use the postMessage bridge for all system interactions. This keeps things secure and prevents crashes.

Secure Environment

Apps are containerized in iframes. No global scripts or styles leak out. Test in isolation, then integrate via bridge.

Kernel Bridge

All requests (e.g., notifications, services) go through window.parent.postMessage({...}, '*'). The kernel handles them.

Persistence

Use localStorage for data. System keys like "system-storage" are shared—respect them to avoid conflicts.

Pro Tip: Debug with console.log in your app. The host console shows kernel logs for bridge events.

Building Your First App

Apps are simple: an HTML file with JS/CSS. Host it on Netlify or GitHub Pages for easy integration. Steps:

  1. Create index.html with your UI/logic.
  2. Add a manifest JSON (see below).
  3. Use postMessage for system features.
  4. Test locally: Open HexagonOS, add your app via store or manual manifest.
  5. Handle lifecycle: Apps can minimize, maximize, close—your code should save state.

Example: A simple counter app.

<!DOCTYPE html>
<html>
<head>
    <title>Counter App</title>
    <style> body { background: #000; color: #fff; } </style>
</head>
<body>
    <h1>Counter: <span id="count">0</span></h1>
    <button onclick="increment()">+</button>
    <script>
        let count = 0;
        function increment() {
            count++;
            document.getElementById('count').textContent = count;
            // Notify system
            window.parent.postMessage({
                type: 'PUSH_NOTIF',
                icon: 'assets/counter-icon.png',
                title: 'Counter',
                message: `Count hit ${count}!`
            }, '*');
        }
    </script>
</body>
</html>

This app increments a counter and sends a notification. Expand with storage, services, etc.

App Manifest (JSON)

The manifest registers your app with the kernel. It's a JSON object defining name, icon, URL, size, etc. System apps are hardcoded, but user apps go in localStorage "apps".

{
  "name": "My App",
  "icon": "assets/icon.png",
  "url": "apps/my-app/index.html",
  "width": "600px",
  "height": "400px",
  "resizable": "true",
  "publish": "com.mydev.app",
  "version": "1.0.0",
  "top": "100px",
  "left": "100px"
  "file_support": ["text","html"]
}

Fields explained:

  • name: Display name.
  • icon: URL to 40x40 PNG.
  • url: Entry HTML (local or remote).
  • width/height: Initial size.
  • resizable: "true" for maximize button.
  • publish: Unique publisher ID.
  • version: Semantic version.
  • top/left: Initial position.

To install: Kernel adds to localStorage on store download.

Sending Notifications

Push global toasts via bridge. They appear on taskbar and log in center.

window.parent.postMessage({
    type: 'PUSH_NOTIF',
    icon: 'assets/alert.png',
    title: 'My App',
    message: 'Task done!'
}, '*');

Icons: Use PNG URLs. Silent apps (e.g., system) skip toasts.

Background Services

Run persistent code even when app is closed. Kernel runs as setInterval.

window.parent.postMessage({
    type: 'PUSH_SERVICE',
    name: 'MyService',
    icon: 'assets/service.png',
    code: 'console.log("Tick");',
    interval: 60000  // ms
}, '*');

Services persist in localStorage "system-external-services". No duplicates—kernel checks.

Desktop Widgets

Widgets are injected directly into the desktop layer. For performance and scalability, use the Registry System. The widget metadata (position, ID) should be stored in system-widgets, while the actual code implementation lives in its own dedicated key.

window.parent.postMessage({
    type: 'PUSH_WIDGET',
    id: 'clock-widget',
    version: '1.4v',
    html: '<div id="clock">...</div><script>...</script>',
    w: '150px', h: '80px',
    x: '20px', y: '20px'
}, '*');

Note: When the system boots, it re-hydrates widgets by executing the stored script tags. Ensure your widget logic is self-contained.

Advanced Storage: Sharding Architecture

To bypass the browser's 5MB localStorage limit, HexagonOS uses a Distributed File System. Developers must understand the split between the "Map" and the "Data".

The Tree (Map)

Stored in system-storage. This is a lightweight JSON array representing the folder hierarchy only. It does not contain file content.

The Shards (Files)

Each folder name becomes its own localStorage key. That key contains an array of file objects: {name, type, content}.

How to Read a File:

  1. Find the folder name in the system-storage tree.
  2. Fetch the data using localStorage.getItem(folderName).
  3. Parse the JSON and find your file by name.
// Example: Reading 'hello.txt' from folder 'my-docs'
const folderData = JSON.parse(localStorage.getItem('my-docs'));
const file = folderData.find(f => f.name === 'hello.txt');
console.log(file.content);

Pro-Tip: Always check if localStorage is full before writing. Large apps should clean up their own shards to keep the OS snappy.

System Storage & Filesystem

Distributed storage to bypass 5MB/key limit. "system-storage" is the tree map; folders are keys holding files.

Format: Array of nested objects for hierarchy.

Initial:
[
  {
    "vs-code": []
  }
]

With subfolders:
[
  {
    "vs-code": [
      { "project-01": [] },
      { "project-02": [] }
    ]
  }
]

Files in folder keys: e.g., localStorage["project-01"] = JSON.stringify([{name: "file.txt", content: "..."}])

Access: Traverse tree, read/write via localStorage.getItem/setItem. Apps like VS Code use this—build similar for photo editors, etc.

Accessing Settings, Themes & Wallpapers

Settings in localStorage "system-settings" (accent, bg, blur). Wallpapers in "system-wallpaper" (url, list, duration).

// Read theme
let settings = JSON.parse(localStorage.getItem("system-settings")) || {};
document.body.style.background = settings.windowBg;

// Update wallpaper
localStorage.setItem("system-wallpaper", JSON.stringify({
  url: "new.jpg",
  wallpaper-list: ["img1.jpg", "img2.jpg"],
  duration: 60  // minutes
}));

Push updates via bridge if needed: type 'UPDATE_SETTING', key/value. Kernel applies globally.

Opening Windows & Apps

Request new windows via bridge. Kernel spawns them.

window.parent.postMessage({
    type: 'OPEN_WINDOW',
    icon: 'assets/icon.png',
    title: 'Sub Window',
    url: 'sub.html',
    width: '400px',
    height: '300px',
    resizable: 'true',
    top: '200px',
    left: '200px'
}, '*');

Use for modals or spawning from services. Kernel handles focus, drag, etc.

Submitting to App Store

Build, test, create manifest JSON. Send to Shino: JSON file + app URL/icon. We'll review/add to store.

Requirements:

  • Secure: No malicious code.
  • Compatible: Use bridge properly.
  • Unique: No duplicates.
  • Versioned: Bump on updates.

Email or DM manifest. Once in store, users install via Web Store app.
